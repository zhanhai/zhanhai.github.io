# 软件架构

**软件架构**是个热门词汇。人们热衷于谈论架构以及相关概念：架构模式、架构风格、响应式/云原生架构、POSA、DDD等，但是大家对架构的一些基础性问题却缺少深入讨论。

就拿最基本的一个问题“什么是软件架构？“来说。我们可以在网络上找到关于架构的各种定义，似乎都符合我们对架构的期望和理解。不同的人也因此接受了不同的架构定义。这种认识上的差异会给架构人员的成长、团队开展架构活动等方面造成困惑。

另外，到底应该如何设计架构呢？对于没有入门的开发人员来说，架构设计充满神秘，完全不知道如何入手。而很多有经验的架构师也不能给出设计方法供人参考和学习，在谈到设计时，通常就是介绍各种模式，要么就是使用一些充满哲学意味的隐喻，让希望学习架构设计的人更加迷茫。

上述问题其实也一直困挠着我。非常幸运的是，我在`Software Architecture in Practice`这本书找到了对这些问题以及其他相关问题的回答。本文就是分享我在学习中的收获和思考。


## 什么是软件架构

> 软件架构是多种结构(`structure`)的集合，通过这些结构，我们可以推导系统的行为和质量属性。而结构由元素以及元素之间的关系构成，元素和关系都有各自特定的属性。

#### 选择元素

1. 设计系统元素时，考虑重要的场景；
2. 用例和场景的选择由架构师负责；
3. 如果场景数目多，可以通过多个迭代持续设计系统元素；
4. 大致规则：10个场景一个迭代，100个场景一个里程碑。
5. 新的需求在设计过程中被提出后，需要重新启动对系统元素的设计以及相关元素的调整；
6. 一般优先使用宽度优先的原则选择元素
